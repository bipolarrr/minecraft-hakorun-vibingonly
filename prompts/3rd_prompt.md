지금 작업은 새 기능 추가가 아니라, 프로젝트 전반의 deprecated API 사용을 최신 API로 마이그레이션하는 유지보수 작업이다.

너는 시니어 Java / Paper 플러그인 유지보수 엔지니어처럼 행동하라.

목표:
- 약 6개 파일에 걸친 deprecated 사용을 최신 API로 교체
- 기존 기능 의미 보존
- 실제 런타임 동작의 확실성 향상
- 빌드 가능한 최종 코드 제시

절대 하지 말 것:
- @SuppressWarnings("deprecation") 로 덮기
- 주석 처리
- 기능 삭제
- 단순 문자열 치환식 수정
- 전체 구조를 불필요하게 갈아엎는 리팩토링

반드시 할 것:
1. 각 deprecated 항목을 개별적으로 분석
2. 최신 대체 API가 무엇인지 명확히 식별
3. 단순 치환으로 끝나는지, 호출 방식 변경이 필요한지 판정
4. 최신 API로 바꿨을 때 의미가 달라질 수 있는 부분 설명
5. import / 타입 / nullability / 예외 / 스레드 문맥까지 같이 수정
6. 최종적으로 파일별 패치와 전체 수정 코드를 제시

특히 중요:
- deprecated 사유가 “이미 동작하지 않음”, “비동기/스레드 안전성 문제”, “더 이상 신뢰 불가”, “이름만 바뀐 대체 API 존재” 중 무엇인지 구분하라.
- Paper/Bukkit 계열이면 최신 권장 방식으로 옮겨라.
- 메서드 하나만 바꾸면 끝나는지, 호출하는 상위 로직도 함께 바꿔야 하는지 검토하라.

원하는 답변 형식:
- deprecated 목록 표준화
- 각 항목별 교체 전략
- 위험도 높은 항목 우선순위
- 파일별 수정 포인트
- 수정된 코드 전체
- 수정 후 검증 체크리스트
